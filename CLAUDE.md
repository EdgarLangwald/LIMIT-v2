## Overview

LIMIT-v2 is the second version of a project with the purpose of stress testing embedding models. The first version sits next to this project, i.e. at 
../LIMIT

## Design Philosophy

All functions in the code must fit into ONLY ONE of three categories:

Dataset creation -> Embedding -> Evaluation. 

This way, multiple tests can be done on multiple models. The embedding has to be able to run locally and on a cluster. And different metrics have to be able to evaluate performance without recomputing embeddings every time.


## Project structure

`src/` is the importable package for the three pipeline stages. `dataset/` contains both the build scripts (checked in) and the generated data (gitignored). Run all commands from the repo root.

```
LIMIT-v2/
├── main.py                       — full-pipeline entrypoint
├── src/                          — importable package (the three pipeline stages)
│   ├── paths.py                  — repo-root-anchored data dirs (DATASET_DIR, EMBEDDINGS_DIR, …)
│   ├── profile_sentences.py      — STAGE 1: sentence category dataclasses (auto-generated)
│   ├── generate.py               — STAGE 1: generate_dataset(n, m, seed)
│   ├── embed.py                  — STAGE 2: embed_dataset, load_model, embed_query
│   ├── evaluate.py               — STAGE 3: evaluate_manually (metrics.py, plotting.py land here next)
│   └── pools/                    — male_names.csv, female_names.csv, family_names.csv
├── tests/                        — test scaffolds (some reference not-yet-built metrics)
├── dataset/
│   ├── build_corpus.js           — Workflow script: rebuilds src/profile_sentences.py (checked in)
│   ├── build_queries.py          — rebuilds qry_final.json benchmark (checked in)
│   ├── query_workflow.js         — Workflow script: LLM step for query generation (checked in)
│   ├── corpus_evaluations/       — per-file Haiku eval cache for build_corpus.js (gitignored)
│   ├── qry_specs/  qry_raw/      — intermediate query-build cache (gitignored)
│   └── qry_final.json            — final query benchmark (gitignored)
├── embeddings/ models/           — cached embeddings and model weights (gitignored)
└── explore_faker.ipynb
```

- `src/paths.py` — single source of truth for data locations; import `DATASET_DIR`/`EMBEDDINGS_DIR`/`MODELS_DIR`/`POOLS_DIR` from here so generated data always lands at the repo root.
- `explore_faker.ipynb` — exploratory notebook for the Faker library
- `src/profile_sentences.py` — auto-generated by `dataset/build_corpus.js`: one dataclass per accepted corpora category, plus name pool dataclasses (loaded from `src/pools/*.csv` at import time). To regenerate: `Workflow({ scriptPath: "dataset/build_corpus.js" })`.
- `src/pools/{male,female,family}_names.csv` — name pools loaded by `profile_sentences.py` at import time (not checked in to avoid 150k-row bloat)
- `src/generate.py` — dataset creation: `generate_dataset(n, m, seed)` builds n person documents each with a Faker intro sentence plus m category-sampled sentences; returns list of dicts with `name`, `gender`, `sentences`, `facts`. Run: `python -m src.generate`.
- `dataset/build_corpus.js` — Workflow script that evaluates every JSON in dariusk/corpora with Haiku and writes `src/profile_sentences.py`. Caches per-file results in `dataset/corpus_evaluations/` (resumable). Run: `Workflow({ scriptPath: "dataset/build_corpus.js" })`.
- `dataset/build_queries.py` — builds the `qry_final.json` benchmark in two stages:
  - `python dataset/build_queries.py --dump-specs --seed S --n N`: samples N *units* (default 100), each = 2 independent *draws*. A draw = 3 weighted-sampled categories (1 item each) + 8 target identities + 8 distinct-category fillers (one per target, each with its own item/template/position). Writes one spec file per unit to `dataset/qry_specs/unit_NNN.json` plus `manifest.json`. No API key needed.
  - (between) `Workflow({ scriptPath: "dataset/query_workflow.js", args: { seed: 42, n_units: 100 } })` spawns 100 Haiku agents (one per unit, writing to `dataset/qry_raw/`). Each agent reads its spec, picks exactly **2 template indices per category**, and writes a **specific** + a **broad** query per draw. Both queries must (a) **describe each item, never name it**, (b) be **selective** — the broad query narrows to a recognizable *sub-family*, never just the bare category, (c) be true under **both** chosen templates of all 3 categories.
  - `python dataset/build_queries.py --render --seed S [--out PATH]`: reads specs + raw, validates, enumerates the **2³ = 8 template combinations → 8 target profiles per draw**, writes `dataset/qry_final.json`: one record per draw with `query_specific`, `query_broad`, `slots`, and the 8 shared `targets`. Each run overwrites `qry_specs/`, `qry_raw/`, `qry_final.json`.
- `src/embed.py` — embedding: `MODELS` dict (13 models, copied from LIMIT v1), `embed_dataset(docs, model_name, dataset_name)` embeds list of person dicts and caches to `embeddings/{dataset_name}/{model_name}_d.npy`; also exports `load_model(model_name)` and `embed_query(query, model_name, model)` for use by evaluate.
- `embeddings/` — cached `.npy` embedding arrays (not checked in)
- `models/` — locally downloaded HuggingFace model weights (not checked in)
- `src/evaluate.py` — evaluation: `evaluate_manually(query, target_name, doc_embs, docs, model_name, k, model)` embeds the query, ranks all documents by cosine similarity, returns `{"top_k": [(rank, name), ...], "target_rank": int|None}`. Metric and plotting modules (e.g. `metrics.py`, `plotting.py`) land here next.
- `tests/` — test scaffolds. Some reference symbols not yet implemented (`src.evaluate.recall_at_k`, a `dataset_creation.build_dataset`) — those mark planned work, not passing tests.
- `main.py` — full pipeline entrypoint: generates dataset, embeds it, loads model, then runs an interactive query/target eval loop. Run: `python main.py` (edit the config block at the top).

### Using `profile_sentences.generate`

The module exposes a single `_Generate` instance called `generate`. Every attribute on it is a callable dataclass that returns a `{"sentence": ..., "facts": {"category": ..., "item": ...}}` dict:

```python
from src.profile_sentences import generate

generate.vegetables(seed=42)                     # male (default)
generate.vegetables(seed=42, gender='female')    # female
```

`gender` accepts `'male'` (default) or `'female'`. Pronouns in templates (`{He}`, `{his}`, `{him}`, `{himself}`, etc.) are resolved by a `_PRONOUNS` dict at call time. `{HisP}`/`{hisP}` resolve to "his"/"hers" (possessive pronoun, not adjective).

Each category dataclass also exposes read-only metadata:

```python
generate.vegetables.pool        # list of items
generate.vegetables.pool_size   # len(pool)
generate.vegetables.templates   # list of template strings
generate.vegetables.frequency   # 0–100, % likelihood this appears in a profile
generate.vegetables.description
generate.vegetables.category_name
```

**All 114 categories** are listed as fields on `_Generate` in [src/profile_sentences.py](src/profile_sentences.py). To see them all: `[f for f in vars(generate)]`.
